8 [Causal Dynamical Triangulations][1] in C++ using the
9 [Computational Geometry Algorithms Library][2] and [Eigen][25]>3.1.0, compiled
10 with [CMake][3] using [Clang][4]/[LLVM][5].
11 Arbitrary-precision numbers and functions by [MPFR][29] and [GMP][30].
12 Option-types via [Boost][47] >= 1.58.0.
13 [Docopt][19] provides a beautiful command-line interface.
14 [Gmock 1.7][6] may be optionally installed in order to build/run unit tests.
15 [Ninja][18] is a nice (but optional) replacement for `make`.
16 Intel's [TBB][37] provides significantly better performance if present (3x+).
17 Follows (mostly) the [Google C++ Style Guide][7] as enforced by [clang-format][46], which
18 you can check by downloading and running the [cpplint.py][8] script:

22 ~~~

55 ### Prerequisites ###
56 ------
57 
58 [CDT++][38] should build on any system (e.g. Linux, MacOS, Windows) with
59 [CMake][14], [CGAL][15], and [Eigen][25] installed. [TBB][37] provides an
60 optional (but significant) speed boost. [Gmock][6] is optional
61 for running the unit tests, and [Ninja][18] is an optional replacement for
62 `make` which provides quick parallel builds of the unit tests.
63 
64 On MacOS, the easiest way to do this is with [HomeBrew][16]:
65 
66 ~~~
67 brew install cmake
68 brew install eigen
69 brew install tbb --c++11
70 brew install ninja
71 brew install cgal --imaging --with-eigen3 --with-lapack
72 ~~~
73 
74 On Ubuntu, you will need an updated versions of gcc and [Boost][47], which you can install via:
75 
76 ~~~
77 sudo add-apt-repository -y ppa:ubuntu-toolchain-r/test
78 # For Ubuntu 14.04
79 # sudo add-apt-repository -y ppa:kzemek/boost
80 sudo apt-get update
81 sudo apt-get install g++-6
82 export CXX="g++-6" CC="gcc-6"
83 ~~~
84 
85 Then you can install the rest of the needed libraries:
86 
87 ~~~
88 # For Ubuntu 16.04 which comes with Boost 1.58.0
89 sudo apt-get install libboost-all-dev
90 # For Ubuntu 14.04
91 # sudo apt-get install boost1.58
92 sudo apt-get install libmpfr-dev
93 sudo apt-get install libgmp3-dev
94 sudo apt-get install cmake
95 sudo apt-get install libeigen3-dev
96 sudo apt-get install libtbb-dev

99 ~~~

103 ### Build ###
104 ------
105 This project uses a separate `build/` directory, which allows you to rebuild the
106 project without cluttering the source code. Thus, download this source code
107 (clone this repo from GitHub or grab a zipped archive [here][17]) and run the
108 following commands in the top-level directory:
109 
110 ~~~
111 mkdir build
112 cd build
113 cmake ..
114 make
115 ~~~
116 
117 (Or run [build.sh][27] if you have [Ninja][18] installed.)
118 
119 This should result in the main program executable, `cdt` in the `build/`
120 directory.
121 
122 If you have [GMock][6] installed and set `GMOCK_TESTS` to **TRUE** (which is the
123 default), the unit test executable, `unittests`, will also be present. See [Tests](#tests) for details.
124 
125 If you are not interested in the unit tests and only want to run the program,
126 set `GMOCK_TESTS` in [CMakeLists.txt][28] to **FALSE**.
127 
128 For some versions of Linux, you may have to build [CGAL][2] from source.
129 Follow the instructions (or their equivalent) given in the install section
130 of the [.travis.yml][39] buildfile.
131 
132 There are enough unit tests that it's worthwhile doing fast parallel builds.
133 [Ninja][18] is just the ticket. It's effectively a drop-in replacement for
134 `make`, and works nicely because [CMake][3] generates the build files.
135 There's quite a difference in speed.
136 
137 Basically, everywhere you see `make` you can type `ninja` instead. Again, see
138 [build.sh][27] for an example.
139 
140 Possible build troubles:
141 -----------------------
142 * While running `build.sh` under linux with `gcc` you may encounter `virtual memory exhausted: Cannot allocate memory`, a common solution for which is to [allocate swap space](http://www.cyberciti.biz/faq/linux-add-a-swap-file-howto/).

144 ### Usage ###
145 ------
146 CDT-plusplus uses [docopt][19] to parse options from the help message, and so
147 understands long or short argument formats, provided the short argument given
148 is an unambiguous match to a longer one. The help message should be instructive:
149 
150 ~~~
151 ./build/cdt --help
152 Causal Dynamical Triangulations in C++ using CGAL.
153 
154 Copyright (c) 2014-2016 Adam Getchell
155 
156 A program that generates d-dimensional triangulated spacetimes
157 with a defined causal structure and evolves them according
158 to the Metropolis algorithm. Specify the number of passes to control
159 how much evolution is desired. Each pass attempts a number of ergodic
160 moves equal to the number of simplices in the simulation.
161 
162 Usage:./cdt (--spherical | --toroidal) -n SIMPLICES -t TIMESLICES [-d DIM] -k K --alpha ALPHA --lambda LAMBDA [-p PASSES] [-c CHECKPOINT]
163 
164 Examples:
165 ./cdt --spherical -n 64000 -t 256 --alpha 1.1 -k 2.2 --lambda 3.3 --passes 1000
166 ./cdt --s -n64000 -t256 -a1.1 -k2.2 -l3.3 -p1000
167 
168 Options:
169  -h --help Show this message
170  --version Show program version
171  -n SIMPLICES Approximate number of simplices
172  -t TIMESLICES Number of timeslices
173  -d DIM Dimensionality [default: 3]
174  -a --alpha ALPHA Negative squared geodesic length of 1-d
175  timelike edges
176  -k K K = 1/(8*pi*G_newton)
177  -l --lambda LAMBDA K * Cosmological constant
178  -p --passes PASSES Number of passes [default: 100]
179  -c --checkpoint CHECKPOINT Checkpoint every n passes [default: 10]
180 ~~~
181 
182 The dimensionality of the spacetime is such that each slice of spacetime is
183 `d-1`-dimensional, so setting `d=3` generates 2 spacelike dimensions and one
184 timelike dimension, with a defined global time foliation. Thus a
185 `d`-dimensional simplex will have some `d-1` sub-simplices that are purely
186 spacelike (all on the same timeslice) as well as some that are timelike
187 (span two timeslices). In [CDT][1] we actually care more about the timelike
188 links (in 2+1 spacetime) and the timelike faces (in 3+1 spacetime).

209 ### Tests ###
210 -----------
211 [GMock][6] is optional, but strongly recommended if you want to make changes to
212 or understand the source code in detail. Building the [GMock][6] `unittests`
213 executable is set by the `GMOCK_TESTS` variable in [CMakeLists.txt][28].
214 
215 To install GMock, you'll need to install GMock and GTest as a shared library. First look at the [README][24] to understand the general idea. On Linux:
216 
217 ~~~
218 git clone https://github.com/google/googletest.git
219 cd googletest/googlemock
220 cmake -DBUILD_SHARED_LIBS=ON .
221 make
222 sudo cp -a include/gmock/ /usr/include/
223 sudo cp -a libgmock_main.so libgmock.so /usr/lib/
224 ~~~
225 
226 GoogleMock is modular, and can use other testing frameworks. Since we want
227 to use GoogleTest, do:
228 
229 ~~~
230 cd ../googletest
231 cmake -DBUILD_SHARED_LIBS=ON .
232 make
233 sudo cp -a include/gtest /usr/include/
234 sudo cp -a libgtest_main.so libgtest.so /usr/lib/
235 ~~~
236 
237 This is scripted in [install-linux.sh][43]. Thanks to [ManuelSchneid3r][42] and
238 [Stack Overflow][40] for the assist.
239 
240 MacOS is very similiar, and the exact steps are scripted in [install-osx.sh][44].
241 
242 
243 Unit tests using GMock are then run (in the `build/` directory) via:
244 
245 ~~~
246 ./unittests
247 ~~~

251 ~~~
252 make test

257 ~~~
258 ninja test
259 ~~~

264 ### Static Analysis ###
265 -----------
266 The [cppcheck-build.sh][35] script runs a quick static analysis using
267 [cppcheck][34].
268 
269 [Clang][4] comes with [scan-build][32] which can run a much more thorough,
270 but slower static analysis integrated with [CMake][3] and [Ninja][18].
271 Simply run the [scan-build.sh][33] script. Note that this
272 script is somewhat fragile, as it depends upon the version of [llvm][5]
273 installed, and links directly to those directories without helpful symlinks
274 to abstract version numbers away.
275 
276 Also, these tools build in **DEBUG** mode. You should probably not then run
277 `unittests` as you will get thousands of lines of debugging output from the
278 tests that create large triangulations. (You could use `--gtest_filter`
279 to run just the tests that you want.)
