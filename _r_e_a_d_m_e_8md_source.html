9 [Causal Dynamical Triangulations][1] in C++ using the
10 [Computational Geometry Algorithms Library][cgal] and [Eigen][eigen]>3.1.0, compiled
11 with [CMake][cmake] using [Clang][clang]/[LLVM][llvm].
12 Arbitrary-precision numbers and functions by [MPFR][mpfr] and [GMP][gmp].
13 Option-types via [Boost][boost] >= 1.63.0.
14 Uses [Docopt][docopt] to provide a beautiful command-line interface, and [Howard Hinnant's date and timezone][date] library for accurate time (both as [subrepos][git-subrepo]).
15 [Gmock 1.8][gmock] may be optionally installed in order to build/run unit tests.
16 [Ninja][ninja] is a nice (but optional) replacement for `make`.
17 Intel's [TBB][tbb] provides significantly better performance if present (3x+).
18 Follows the [CppCore Guidelines][guidelines] as enforced by [clang-tidy][tidy], which you can run using the
19 [clang-tidy.sh][8] script:
20 
21 ~~~
22 ./clang-tidy.sh
23 ~~~

25 (Or use your favorite linter plugin for your editor/IDE.)

56 ### Prerequisites ###
57 ------
58 
59 [CDT++][38] should build on any system (e.g. Linux, MacOS, Windows) with
60 [CMake][14], [CGAL][cgal], [Boost][boost], [MPFR][mpfr], and [Eigen][eigen] installed.
61 [TBB][tbb] provides an optional (but significant) speed boost. [Gmock][gmock] is
62 optional (but recommended) for running the unit tests, and [Ninja][ninja] is an
63 optional replacement for `make` which provides quick parallel builds of the
64 unit tests.
65 
66 On MacOS, the easiest way to do this is with [HomeBrew][homebrew]:
67 
68 ~~~
69 brew upgrade cmake
70 brew install ninja
71 brew upgrade boost
72 brew install eigen
73 brew install tbb --c++11
74 brew install cgal --imaging --with-eigen3 --with-lapack
75 ~~~
76 
77 On Ubuntu, you will need an updated versions of [clang][clang] or [gcc][gcc],
78 [CMake][cmake], and [Boost][boost], which is scripted in [.travis.yml][39].
79 With Howard Hinnant's [date][date] library, you may also need to install
80 `libcurl-dev`, which is a virtual package with many flavors, OpenSSL being
81 the most tried and tested.

83 ### Build ###
84 ------
85 This project uses a separate `build/` directory, which allows you to rebuild the
86 project without cluttering the source code. Thus, download this source code
87 (clone this repo from GitHub or grab a zipped archive [here][17]) and run the
88 following commands in the top-level directory:
89 
90 ~~~
91 mkdir build && cd build
92 cmake -DCMAKE_BUILD_TYPE=Release ..
93 cmake --build .
94 ~~~
95 
96 (Or run [build.sh][27] if you have [Ninja][ninja] installed.)
97 
98 This should result in the main program executable, `cdt` in the `build/`
99 directory, along with several others.
100 
101 * `cdt-gv` converts output files to GeomView format for visualization
102 * `cdt-opt` is a simplified version with hard-coded inputs, mainly useful for
103 debugging and scripting
104 
105 If you build [GMock][gmock] unit tests, the executable
106 `unittests` will also be present. See [Tests](#tests) for details.
107 
108 For some versions of Linux, you may have to build [CGAL][cgal] from source.
109 Follow the instructions (or their equivalent) given in the install section
110 of the [.travis.yml][39] build file.
111 
112 There are enough unit tests that it's worthwhile doing fast parallel builds.
113 [Ninja][ninja] is just the ticket. It's effectively a drop-in replacement for
114 `make`, and works nicely because [CMake][cmake] generates the build files.
115 There's quite a difference in speed. CMake also [abstracts away the build][47]
116 tool nicely. Again, see [build.sh][27] for an example.
117

119 -----------------------

122 which is to
123 [allocate swap space](http://www.cyberciti.biz/faq/linux-add-a-swap-file-howto/).

125 ### Usage ###
126 ------
127 CDT-plusplus uses [docopt][docopt] to parse options from the help message, and so
128 understands long or short argument formats, provided the short argument given
129 is an unambiguous match to a longer one. The help message should be instructive:
130 
131 ~~~
132 ./build/cdt --help
133 Causal Dynamical Triangulations in C++ using CGAL.
134 
135 Copyright (c) 2014-2017 Adam Getchell
136 
137 A program that generates d-dimensional triangulated spacetimes
138 with a defined causal structure and evolves them according
139 to the Metropolis algorithm. Specify the number of passes to control
140 how much evolution is desired. Each pass attempts a number of ergodic
141 moves equal to the number of simplices in the simulation.
142 
143 Usage:./cdt (--spherical | --toroidal) -n SIMPLICES -t TIMESLICES [-d DIM] -k K --alpha ALPHA --lambda LAMBDA [-p PASSES] [-c CHECKPOINT]
144 
145 Examples:
146 ./cdt --spherical -n 64000 -t 256 --alpha 0.6 -k 1.1 --lambda 0.1 --passes 1000
147 ./cdt --s -n64000 -t256 -a.6 -k1.1 -l.1 -p1000
148 
149 Options:
150  -h --help Show this message
151  --version Show program version
152  -n SIMPLICES Approximate number of simplices
153  -t TIMESLICES Number of timeslices
154  -d DIM Dimensionality [default: 3]
155  -a --alpha ALPHA Negative squared geodesic length of 1-d
156  timelike edges
157  -k K K = 1/(8*pi*G_newton)
158  -l --lambda LAMBDA K * Cosmological constant
159  -p --passes PASSES Number of passes [default: 100]
160  -c --checkpoint CHECKPOINT Checkpoint every n passes [default: 10]
161 ~~~
162 
163 The dimensionality of the spacetime is such that each slice of spacetime is
164 `d-1`-dimensional, so setting `d=3` generates 2 spacelike dimensions and one
165 timelike dimension, with a defined global time foliation. Thus a
166 `d`-dimensional simplex will have some `d-1` sub-simplices that are purely
167 spacelike (all on the same timeslice) as well as some that are timelike
168 (span two timeslices). In [CDT][1] we actually care more about the timelike
169 links (in 2+1 spacetime) and the timelike faces (in 3+1 spacetime).

191 ### Tests ###
192 -----------
193 [GMock][gmock] is optional, but strongly recommended if you want to make changes to
194 or understand the source code in detail. Building the [GMock][gmock] `unittests`
195 executable is set by the `GMOCK_TESTS` variable in [CMakeLists.txt][28], or
196 at the command line using:
197 
198 ~~~
199 cmake -DGMOCK_TESTS:BOOL=ON -DCMAKE_BUILD_TYPE=Debug ..
200 ~~~
201 
202 GMock is setup and installed by CMake using the [technique described here.][cmake-gtest] Consult [CMakeLists.txt][28] for details.
203 
204 Unit tests using GMock are then run (in the `build/` directory) via:
205 
206 ~~~
207 ./unittests
208 ~~~
209 
210 You can also run function tests (using [CTest][ctest]) via:
211 
212 ~~~
213 cmake --build . --target test
214 ~~~
215 
216 In addition to the command line output, you can see detailed results in the
217 `build/Testing` directory which is generated thereby.

219 ### Static Analysis ###
220 -----------
221 The [cppcheck-build.sh][35] script runs a quick static analysis using
222 [cppcheck][cppcheck].
223 
224 [Clang][clang] comes with [scan-build][32] which can run a much more thorough,
225 but slower static analysis integrated with [CMake][cmake] and [Ninja][ninja].
226 Simply run the [scan.sh][33] script.
227 
228 Also, these tools build in **DEBUG** mode. You should probably not then run
229 `unittests` as you will get thousands of lines of debugging output from the
230 tests that create large triangulations. (You could use `--gtest_filter`
231 to run just the tests that you want.)
232 
233 One of the [Travis-CI][travis] jobs runs [Valgrind][valgrind]; be sure to look at the
234 results to ensure you're not leaking memory.
