How to Contribute

and///

First, thank you!

Writing esoteric scientific software can be it's own reward, but it's not for the faint of heart.

Second, here are some simple guidelines that will make it easier on me to process and accept your contributions.

  1. Fork the repository
  2. Install GMock/GTest. See install-linux.sh or install-osx.sh for details.
  3. Write a unit test for your proposed contribution. Unit tests go in the unittests directory and are named <YourContribution>Test.cpp, so that they can be automatically built. All proposed features of your contribution should have a corresponding test in <YourContribution>Test.cpp. Consult the https://github.com/google/googletest/blob/master/googlemock/README.md "GMock" and https://github.com/google/googletest/blob/master/googletest/docs/Primer.md "GTest" documentation if you are unsure, or consult existing tests for examples. Prefer predicate assertions because their output is much easier to understand and useful for debugging.
  4. I highly recommend writing your tests first, before your contribution, as this helps to think about how the rest of the program will use your functions and/or classes. This Test-Driven Development has saved me quite a lot from various mistakes.
  5. Project source code goes into the src directory, generally as header files named <YourContribution>.h. This makes inclusion into tests and the main program easy and modular.
  6. Don't forget documentation! It's helpful if you state explicitly what your functions and/or classes do. I use Doxygen to automatically build documentation, so using /// @param[] is helpful and easy. Consult existing code for examples.
  7. Commit your changes with a clear, well-written commit message. I am guilty of not always writing the clearest commit messages in this project, and will do better. For the longest time, however, no one but me actually read them. When I contribute to other projects, I definitely try to have useful, clear commit messages and you should, too.
  8. Open a pull request against the main repository. Travis-CI will test it against Linux/gcc-5 and MacOS/clang, so ensure that your code compiles on both platforms.
  9. I will get to your change as soon as I can. You will receive proper credit for your contributions both in the code and any resulting scientific papers.

Style Guide

This project generally follows the Google C++ Style Guide, except when it doesn't. I use C++14 constructs freely and cheerfully ignore advice from the guide to do otherwise. These annoying warnings can be turned off on individual lines using NOLINT. Here's an example (using C++14 automatic return type deduction):

1 {C++}
2 auto fix_timeslices(T&& universe_ptr) { // NOLINT

The cpplint.py script is helpful in enforcing this, and most editors/IDEs have a C++ linter plugin to use this.
